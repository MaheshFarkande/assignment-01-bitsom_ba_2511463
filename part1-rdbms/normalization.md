
## Anaomaly Analysis
1) Insert Anomaly (can’t insert a fact about one entity without another)  - Row identified by order_id = ORD1185 , the product Webcam exists only because there is an order row for it
2) Update Anaomaly(same fact stored multiple times → must update in many places, risk inconsistency) -Row identified by order_id = ORD1114 shows SR01 address as Nariman Point. But in other rows for the same SR01, the address is stored differently - Row identified by order_id = ORD1180 shows Nariman Pt
3) Delete Anomaly (deleting one row unintentionally deletes other facts) -The Webcam product (P008) appears (in the provided data) in the order row identifier id: order_id = ORD1185, deletion of this row will delete Webcam product details.

## Normalization Justification
While a single flat table may look simpler at first glance, our own dataset shows why this breaks down in practice and why normalization is not over‑engineering. In orders_flat.csv, the same real‑world facts are duplicated across many rows, which has already led to inconsistencies—e.g., the same sales rep (SR01) appears with two different office address spellings (“Nariman Point” vs “Nariman Pt”), proving that updates are error‑prone when the same attribute is stored repeatedly. The design also creates insert and delete anomalies: products only exist if an order exists (you cannot add a new product without fabricating an order), and deleting an order can remove the only stored definition of a product. Normalizing to 3NF fixes these concrete issues by storing customers, products, and offices once and referencing them by keys, so updates happen in one place, products can exist independently of orders, and deleting an order does not erase unrelated business facts. In short, for this dataset, normalization is the minimum structure required to preserve data correctness as the system evolves, not unnecessary complexity.
